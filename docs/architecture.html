Section 01

System Architecture

Five horizontal layers: mobile client, REST transport, Express API server, external third-party services, and the persistence + background-job layer. All routes use Prisma ORM to read/write SQLite.

Section 02

Key Data Flows

Four critical paths through the system — from initial registration to real-time AI guidance during a live shoot.

1
Authentication & Session Management
01Register: mobile sends email + password (min 8 chars) to POST /auth/register
02Server validates via Zod, checks email uniqueness, hashes password with bcrypt (cost 10)
03Server signs accessToken (15 min) and refreshToken (7 days) with separate JWT secrets
04Mobile stores both tokens in expo-secure-store, sets isAuthenticated: true in Zustand
05On every request, axios interceptor injects Authorization: Bearer {accessToken} header
06On 401, interceptor POSTs refreshToken to /auth/refresh, retries original request with new token
07App restart: AuthGuard reads stored token, fetches /profile to confirm validity, routes appropriately
2
Style Profiling — Quiz → AI Inference
01Step 1: user selects genres (landscape, astro…) — stored in Zustand styleStore client-side only
02Step 2: user picks moods → POST /quiz/setup persists genres + moods to StyleProfile in DB
03Step 3: GET /quiz/photos?round=N — server reads genres from DB, queries Unsplash with pre-tagged searches, caches result 7 days
04User taps like/skip on 4 photos → POST /quiz/response per photo saves PhotoPreference rows with tags + selected boolean
05Repeats 10 rounds (40 photos total) — progress bar advances each round
06POST /quiz/complete — server responds immediately, fires async buildStyleProfile()
07Claude Haiku analyses selected vs rejected tag sets, returns 14-dim style vector (0.0–1.0) + one-sentence summary
08Confidence score = 0.3 + selected_count × 0.05, capped at 0.95. Stored in StyleProfile. Client polls /profile/style.
3
Real-Time Camera Guidance
01Client-side only: useSunPosition(lat, lng) calls SunCalc every render — zero API cost, updates continuously
02SunCalc computes azimuth, altitude, isGoldenHour, isBlueHour, sceneLabel from current time + GPS
03TanStack Query fires GET /guidance/scene?sunAzimuth=…&sunAltitude=…&sceneLabel=… with staleTime: 15min
04Server builds cache key from altitude bucket (golden/blue/other), sun quadrant (0–7 of 45°), and user's top style tag
05Cache hit: returns in <1ms. Miss: Claude Haiku generates 15-word composition tip personalised to style vector
06Response includes lightQuality, styleMatchScore (dot product of scene tags vs vector), compositionTip
07Camera screen shows tip in <GuidanceCard>, sun direction in <SunDirectionIndicator>, composition grid overlay
08Session tracked: POST /guidance/session/start on camera ready → POST /session/:id/end on shutter, stores overlaysShown
4
Spot Discovery & DB Geocaching
01useLocation() requests foreground GPS permission. On deny: default London coords + permission banner shown in UI
02User can type a city name → Location.geocodeAsync(city) (platform geocoding, no API key) → updates active coords
03TanStack Query fires GET /spots/nearby?lat=…&lng=…&radius=3000 on location ready
04DB cache check: server queries Spot rows within ±0.05° and expiresAt > now. If ≥5 rows → return immediately
05Cache miss: calls Google Places nearbysearch API (tourist_attraction type), upserts up to 15 spots with 24h expiry
06Server maps Google Place types to style tags (park → landscape/golden_hour) and infers best shooting times
07Client computes Haversine distance from active location to each spot, sorts nearest-first, renders list + MapView markers
08Tapping a list card calls mapRef.animateToRegion() to zoom the map. Spot view recorded as UserSpotView row.

Section 03

Design Tradeoffs

Every architectural choice involves a tradeoff. These are the key decisions made and the reasoning behind each.

Decision Chosen Alternative Reasoning Impact & Limitation
Database engine SQLite + Prisma PostgreSQL / MySQL Zero infrastructure cost for MVP. Single-file DB, no server to manage, instant setup. Prisma ORM means migration to Postgres later is a one-line config change. Cannot handle concurrent writes under load. File-level locking means horizontal scaling is impossible. Not suitable beyond ~5k DAU.
Caching layer NodeCache (in-memory) Redis / Memcached No additional service to deploy or pay for. Sufficient for single-process server. All cache data is re-derivable from source APIs. Cache is wiped on every server restart. Cannot share cache across multiple Node.js processes. Redis would enable horizontal scaling and cache persistence.
AI model tier Claude Haiku 4.5 Sonnet / Opus ~$0.0001/call vs ~$0.003/call for Sonnet. Style inference and guidance tips are well-structured JSON tasks that don't need Sonnet's reasoning depth. 512 token output is sufficient. Guidance tips may lack nuance for edge-case scenes. Style inference quality bounded by model capability. Can upgrade by changing a single model string.
Sun position calculation SunCalc client-side Server-side API SunCalc is a pure math library — zero API calls, zero latency, works fully offline. Runs every render update. A server call would add 100–200ms latency per frame. Slightly increases JS bundle size. All users share the same calculation algorithm (no server-side optimisation possible). Accuracy relies on device GPS precision.
Auth mechanism Stateless JWT Session + DB store No server-side session state means any server instance can validate tokens. 15-min access token limits blast radius of leakage. Refresh token enables silent renewal. Cannot immediately revoke a compromised access token (must wait 15 min). Refresh tokens in SecureStore are a hard-to-rotate secret. No built-in CSRF protection.
Style inference timing Async fire-and-forget Synchronous (user waits) Claude inference takes 1–3 seconds. Making the user wait on quiz completion would feel slow. Instead, /quiz/complete responds immediately and inference runs in background. Client must poll /profile/style to detect when inference is ready. If server crashes mid-inference, the job is lost (no persistent job queue).
Mobile framework Expo Managed Bare React Native Expo Go enables instant test on real device without Xcode/Android Studio builds. SDK abstractions (expo-camera, expo-location) reduce native code. OTA updates possible. Limited native module support — certain camera features (actual photo capture) require an Expo development build. EAS build pipeline adds CI complexity at scale.
Spots storage strategy DB as geocache Always call Places API Google Places is billed per request. Caching spots in SQLite with 24h expiry means repeated lookups in the same area cost nothing. Reduces latency from ~300ms to <5ms. Stale data risk: spots that close or change within 24h won't update. Cache hit radius (±0.05°) is coarse and may miss nearby spots just outside the bounding box.
Style vector representation 14-dim float array (JSON) Normalised categorical tags Continuous 0.0–1.0 scores enable nuanced matching (e.g. 0.8 golden_hour vs 0.3) rather than binary presence. Dot-product styleMatchScore is trivial to compute. JSON stored in SQLite TEXT column. No type safety on the JSON column — a schema mismatch silently returns wrong scores. Vector dimensions are fixed at design time; adding a new dimension requires a migration.

Section 04

Technical Insights

Non-obvious implementation patterns that significantly influence performance, cost, or user experience.

🗝
Guidance Cache Key Design
The guidance cache key guidance:{altBucket}:q{quadrant}:{topTag} is designed for maximum reuse. Sun azimuth is bucketed into 8 quadrants (45° each) and altitude into three buckets (golden/blue/other). Two users pointing at different compass directions in the same lighting condition share the same cached tip. This drives the hit rate above 80% during golden hour when demand spikes.
caching · cost optimisation
🧮
14-Dimensional Style Vector
The style vector intentionally uses non-normalised independent dimensions so a photographer can score high on both golden_hour and blue_hour simultaneously. The styleMatchScore in guidance responses is a simple dot product between the scene's active tag set and the user's vector — O(14) computation, zero API call. Confidence is capped at 0.95 to always leave room for improvement.
AI · feature engineering
🌍
DB as a Persistent API Cache
Google Places results are upserted into SQLite with an expiresAt timestamp. The server checks this cache before every Places API call. On a popular route (e.g. central London), the first user populates the cache; all subsequent users that day pay zero API cost and get sub-5ms responses. This pattern reduces Places API spend by an estimated 80–90% in dense urban areas.
caching · cost · latency
⚡
Zero-Cost Sun Position
SunCalc runs entirely on the client device using pure trigonometry — no network request, no API key, no rate limit. It updates on every component render cycle, meaning the golden-hour banner reacts in real time as the sun crosses the threshold. A server-side approach would introduce 100–400ms round-trip latency and cost per user per session.
client-side · free computation
🔄
Fire-and-Forget Inference Pattern
POST /quiz/complete calls buildStyleProfile() with .catch() but never awaits it before responding. The HTTP response goes back to the client in <50ms while Claude processes in background for 1–3 seconds. The risk is that a server crash between response and inference completion leaves the user with DEFAULT_VECTOR until next login triggers a re-check. A persistent job queue (BullMQ/Redis) would fix this.
async · resilience tradeoff
🛡
Layered Fallback Strategy
Every external API call has an explicit fallback: Unsplash failure → gradient placeholder cards; Google Places failure → 3 hardcoded mock spots; Claude failure → DEFAULT_VECTOR + rule-of-thirds tip; OpenWeather failure → clear 20°C defaults; GPS denied → London coordinates + UI banner. The app never crashes on any single API outage — it degrades gracefully at each layer independently.
resilience · UX continuity
📊
Embedded Cost Monitoring
Every Anthropic and Google API call writes an APICallLog row with latencyMs, statusCode, tokensUsed, and a costEstimate. A DailyMetricRollup cron aggregates these nightly. The GET /telemetry/metrics dashboard exposes quiz completion rate, notification open rate, and total AI spend — observable from day one without a third-party APM tool.
observability · cost tracking
🔐
Token Lifecycle Management
The axios response interceptor handles the full token lifecycle transparently: on 401, it reads the refresh token from SecureStore, calls /auth/refresh, updates storage, then retries the original request — all without any user interaction. The original request queue is not preserved (concurrent 401s could issue multiple refresh calls), but this is acceptable for a mobile app with typically one in-flight request.
auth · silent renewal
🗺
Client-Side Haversine Sorting
Rather than asking the server to sort by distance (which would require the server to know the user's exact coordinates for every request), the client computes Haversine distances from the active location to every returned spot and sorts the list locally. This keeps the server response stable and cache-friendly, while the client adapts the display order dynamically — including when the user manually overrides their city.
client computation · cache stability

Section 05

Risk Management

Known risks ranked by likelihood × impact. Mitigations are proposed for each.

Risk Likelihood Impact Severity Description Mitigation
SQLite write contention High High Critical SQLite uses file-level locking. Multiple concurrent writes (telemetry batch + notification scheduler + user request) will queue and may timeout under load above ~50 concurrent users. Migrate to PostgreSQL via single Prisma datasource change. Use WAL mode in SQLite (PRAGMA journal_mode=WAL) as immediate stopgap to allow concurrent reads with one writer.
No job persistence queue Med High High Style inference is fire-and-forget. If the server restarts mid-inference or Claude times out, the user is silently left with DEFAULT_VECTOR. No retry, no dead-letter queue, no visibility. Add BullMQ + Redis for a persistent job queue. Fall back to polling: client checks styleProfile.confidence > 0 on app foreground; if still 0 after quiz, re-trigger inference.
Single Anthropic API key Low High High If the Anthropic API key is rate-limited, revoked, or Anthropic has an outage, all guidance tips and style inference calls fail. No fallback API key rotation exists. Fallback logic already returns DEFAULT_VECTOR and rule-of-thirds tip. Add key rotation with env-var array. Implement exponential backoff. Monitor via APICallLog statusCode = 429/500.
Google Places cost spike Med High High Google Places Nearby Search costs $0.032/request. A viral moment or bot could exhaust the free tier ($200/month credit) rapidly. No rate limiting on /spots/nearby currently. Add per-user rate limiting (express-rate-limit): max 20 spot queries/hour. Enforce minimum 1km radius. Add Google Cloud budget alert at 50% of monthly limit. Pre-seed popular cities.
JWT refresh token theft Low High Med expo-secure-store protects tokens from casual extraction, but a jailbroken device or memory exploit could read them. 7-day refresh token gives a long attack window with no revocation. Implement refresh token rotation (issue new refresh token on each use). Store refresh token hash in DB with revokedAt field. Reduce refresh TTL to 24–48h. Add device fingerprint claim.
No API rate limiting High Med Med All authenticated endpoints have no per-user rate limiting. A buggy client, malicious actor, or retry storm could flood the server and upstream APIs with requests. Add express-rate-limit globally (e.g. 300 req/15min per user). Apply stricter limits on expensive endpoints: /guidance/scene (30/min), /quiz/response (120/min).
In-memory cache lost on restart High Low Low NodeCache is in-process. Every server deploy or crash flushes all cached guidance tips, weather data, and quiz photos — causing a burst of upstream API calls on cold start. Pre-warm cache on startup for top N city coordinates. Migrate to Redis for durable cache that survives restarts. Add circuit-breaker to prevent thundering-herd on cold start.
No DB backup strategy High High Critical The SQLite file prisma/visionweave.db has no automated backup. A disk failure, accidental rm, or corrupted file would permanently destroy all user accounts, style profiles, and spots. Add cron to copy DB file to object storage (S3/R2) every 6 hours. Enable SQLite WAL for crash safety. On migration to Postgres, use managed DB with automated daily snapshots.
Expo Go camera limitations High Med Med Actual photo capture (CameraView.takePictureAsync) does not work in Expo Go — only in development or production builds. The shutter button shows a placeholder alert, not a real capture. Build an Expo Development Build (eas build --profile development) for camera testing. Document this limitation clearly. Photo capture is non-critical for the MVP guidance overlay use case.
HTTP in development (no TLS) Med Low Low API_BASE_URL is http://localhost:3001. On a physical device over LAN, traffic (including JWT tokens) is sent unencrypted. Trivially sniffable on shared networks. Development only — acceptable for MVP. For TestFlight/production: deploy behind HTTPS reverse proxy (nginx + Let's Encrypt or Cloudflare Tunnel). Update API_BASE_URL to HTTPS endpoint.
